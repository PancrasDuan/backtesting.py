# Backtesting.py 架构学习文档设计

## 1. 背景与目标

本次交付以仓库基线 `cadcbe2` 和 `ARCHITECTURE_LEARNING_GOALS.md` 为依据，生成两份面向源码学习者的中文文档：

- `BACKTESTING_ARCHITECTURE_LEARNING_GUIDE.md`：适合在仓库中持续阅读、检索、批注和跟随源码跳转的完整教材。
- `BACKTESTING_ARCHITECTURE_LEARNING_GUIDE.html`：适合按学习阶段探索的交互式学习地图。

两份文档共享同一知识结构，并独立覆盖目标文件的全部重点。Markdown 承担深度解释，HTML 强化架构关系、过程变化、进度管理和主动回忆。

完成交付后，读者应能独立说明 Backtesting.py 如何把 OHLC(V) 数据、`Strategy` 子类和回测参数转化为订单、成交、权益曲线与统计结果，并能判断其研究型回测边界。

## 2. 受众与阅读假设

- 受众具有 Python 基础，能阅读类、属性、迭代器、NumPy 与 Pandas 代码。
- 不假设读者熟悉事件驱动回测、撮合引擎或账户核算。
- 学习以源码架构为主，不以新增策略、修改框架或介绍通用量化理论为目标。
- 所有源码链接和行号均以当前仓库基线为准；正文优先使用稳定的符号名，降低后续行号漂移造成的影响。

## 3. 共同内容架构

两份文档都按下列知识骨架组织：

1. 学习导航：先修知识、建议节奏、阅读方式和完成标准。
2. 系统全景：模块职责、核心对象所有权、公开 API 到内部模块的边界。
3. 一次回测的完整旅程：构造、`init()`、预热、逐 bar 循环、撮合、策略决策、收尾与统计。
4. 七阶段源码精读：公开 API、时间推进、订单状态、撮合、统计、优化并行、测试规格。
5. 四条关键调用链：指标、下单成交、平仓、结果绘图。
6. 交易推演实验：带 `spread`、`commission`、`SL`、`TP` 的订单生命周期及同 bar 歧义。
7. 架构判断：状态所有权、不变量、模块边界、警告和失败条件。
8. 能力边界与迁移：从研究回测走向个人量化或实盘系统所缺失的能力。
9. 学习验收：九项验收清单、口述题、源码定位题和推演题。
10. Fork 与上游同步：远端约定及学习分支建议。

每个学习阶段使用统一模板：

- 本阶段要回答的问题
- 推荐源码顺序
- 核心机制解释
- 可视化
- 源码观察点
- 常见误解与边界
- 动手推演或自测
- 完成检查

## 4. Markdown 文档设计

### 4.1 表现形式

- 使用 Mermaid 绘制模块依赖图、对象所有权图、主循环时序图、订单状态图、撮合决策流和优化并行图。
- 使用表格表达模块职责、订单类型、价格规则、统计来源和研究系统/实盘系统边界。
- 使用引用块突出关键结论，使用检查框形成可执行学习路线。
- 使用短小源码摘录解释机制；摘录只保留与当前概念直接相关的代码，并链接回完整源码。
- 提供目录和阶段间的前后依赖，让文档既能顺读，也能作为速查手册。

### 4.2 深度要求

Markdown 是事实与推理的主载体。它必须明确解释：

- `Strategy.init()` 与 `Strategy.next()` 的数据可见性差异，以及绕过 `_Data` 保护造成未来数据泄漏的方式。
- `Backtest.run()` 中 `data._set_length(i + 1)`、指标切片、`broker.next()` 和 `strategy.next()` 的严格顺序。
- `Order`、`Trade`、`Position` 的区别，及 `_Broker` 作为交易状态唯一事实来源的意义。
- 市价、限价、止损市价、止损限价和 contingent order 的触发与定价规则。
- spread、commission、margin、杠杆、持仓、现金和权益之间的关系。
- 同一 OHLC bar 内价格路径不可知带来的撮合歧义。
- `_trades`、`_equity_curve`、公开统计字段和绘图输入的来源。
- 网格、随机网格、SAMBO、共享内存和 worker 返回值边界。
- `MultiBacktest` 与共享资金多资产组合引擎的区别。

## 5. HTML 文档设计

### 5.1 页面结构

HTML 使用单文件、无构建步骤的实现，确保双击即可阅读：

- 顶部显示整体学习进度和当前阶段。
- 侧边或顶部阶段导航在窄屏下自动重排。
- 主内容区按七个阶段和三个综合章节组织。
- 每阶段包含概念摘要、源码入口、关键图、易错点、自测和完成按钮。
- 综合区提供四条调用链切换、订单生命周期步进器、架构边界对照和最终验收清单。

### 5.2 交互

- 阶段导航：在不重新加载页面的情况下切换学习章节。
- 折叠卡片：按需展开源码解释、误区和答案，减少首屏认知负担。
- 调用链切换：在同一画布中突出指标、下单、平仓和结果链。
- 订单生命周期步进：展示“创建订单 → 下一 bar 撮合 → 开仓 → 创建 SL/TP → 触发 → 平仓结算”的状态变化。
- 自测题：先作答或思考，再展开参考答案。
- 学习进度：使用浏览器 `localStorage` 保存阶段完成状态；存储不可用时退化为当前会话状态，不影响阅读。
- 主题与无障碍：适配明暗主题、键盘操作和移动端布局，交互不依赖颜色作为唯一编码。

### 5.3 技术边界

- 产物为自包含 HTML，不请求远程接口，不依赖服务端或构建工具。
- JavaScript 只负责导航、展开/收起、状态切换和进度持久化，不承载事实数据计算。
- HTML 内的架构内容来自与 Markdown 相同的内容清单，但按交互阅读方式改写，而不是机械复制。
- 禁用 JavaScript 时，所有核心章节仍按普通文档顺序可读；交互只是增强层。

## 7. 内容正确性与失败处理

- 对学习目标中的每一项建立覆盖矩阵，验证其至少在两份正式文档中各出现一次。
- 对重要机制直接核对当前源码和相关测试，避免只复述目标文件。
- 对 OHLC 内部路径、成交顺序、统计年化假设等不确定或模型化行为，明确标注“框架假设”或“模型限制”。
- 源码链接使用仓库相对路径；若浏览器不能从 HTML 打开本地相对链接，正文仍展示文件和符号名，保证可定位。
- Mermaid 渲染失败时，Markdown 中相邻文字和表格仍足以表达核心含义。
- HTML 的脚本错误不能隐藏正文；核心内容默认存在于 DOM 中，脚本只改变呈现状态。

## 8. 验证方案

### 8.1 内容验证

- 建立目标覆盖表，逐项核对总目标、七阶段、四条调用链、架构不变量、项目边界、九项验收和 Fork 同步。
- 检查所有核心类、方法和模块名称能在当前仓库中找到。
- 用最小行情示例手工复核订单时序、成交价、spread、commission、保证金和 SL/TP 状态变化。

### 8.2 Markdown 验证

- 检查标题层级、目录锚点、代码围栏、Mermaid 围栏、相对链接和表格结构。
- 检查所有 Mermaid 图的语法和节点命名，避免渲染器不兼容字符。
- 检查文档可在不渲染 Mermaid 的情况下顺畅阅读。

### 8.3 HTML 验证

- 验证 HTML 语法、脚本无未定义标识符，所有选择器都有对应元素。
- 在浏览器中检查桌面与窄屏布局、明暗主题、键盘焦点和内容溢出。
- 逐项操作阶段导航、折叠卡片、调用链切换、订单步进、自测答案和进度保存。
- 刷新页面确认进度恢复；模拟 `localStorage` 不可用确认退化路径。
- 确认禁用 JavaScript 后核心内容仍可阅读。

## 9. 非目标

- 不修改 Backtesting.py 的运行代码、测试或公开 API。
- 不把学习文档扩展成一般性的量化交易教程或策略库。
- 不引入前端框架、打包器、服务端或外部数据源。
- 不承诺对未来上游版本自动保持正确；文档明确绑定基线 `cadcbe2`。

## 10. 完成标准

交付满足以下条件时视为完成：

- 两份文档均存在且可独立阅读。
- `ARCHITECTURE_LEARNING_GOALS.md` 的所有重点均有明确覆盖位置。
- Markdown 含完整七阶段路线和至少六类架构可视化。
- HTML 包含阶段导航、至少三种实质性交互和可恢复的学习进度。
- 两份文档对关键机制的表述与当前源码、测试一致。
- HTML 已完成实际渲染与交互检查，Markdown 已完成结构和链接检查。
