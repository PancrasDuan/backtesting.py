学习路线

总览 · 先看所有权，再看调用

一个回测，是六层协作的一次状态旅程

输入是一份 OHLC(V) DataFrame、一个 Strategy 子类和参数；输出是订单与交易历史、完整权益曲线、_Stats 和图表。关键不是对象有多少，而是谁拥有可变事实。

  1. 公开入口
    Backtest.run
  2. 策略层
    Strategy.I
  3. 时间窗口
    _Data / _Indicator
  4. 交易真相
    _Broker
  5. 结果
    compute_stats / _Stats
  6. 并行包装
    Pool / MultiBacktest

策略表达意图

Strategy 读取当前窗口，通过 buy()/sell() 创建订单；它不维护账户真相。

Broker 拥有状态

_Broker 独占 orders、活动 trades、closed trades、cash 与 equity。Position 只是汇总视图。

统计只消费结果

compute_stats 从交易和权益生成结果；_plotting 读取结果而不参与成交。

并行有明确边界

SharedMemoryManager 传数据，Pool 执行参数批次；最佳参数回到主进程完整复跑。

一次 run 的九个动作

  1. 校验并保存输入；每次 run() 新建 _Data、_Broker 与策略实例。
  2. 校验策略参数，调用一次 init()，让 Strategy.I 全量预计算指标。
  3. _Data._update() 重建列数组，识别指标与 warmup。
  4. 每根 bar 先 _set_length(i + 1)、同步指标切片，再 broker.next()，最后 strategy.next()。
  5. finalize_trades 可在最后 OHLC 上强制处理剩余平仓单。
  6. 补齐完整权益，交给 compute_stats。
第一不变量：交易状态的唯一事实来源是 _Broker。Order、Trade、Position 和 Strategy 都不能另存一套“账户真相”。

这使当前 strategy.next() 新建的默认市价单只能在下一次 broker tick 处理，通常按下一 bar Open 成交，从而保留 bar 级因果顺序。trade_on_close 只改变下一次撮合采用前一 Close 和时间索引，并非同步成交。

Backtest.run 负责装配，Strategy.I 负责声明指标

核心机制

Backtest.__init__ 是配置阶段：浅复制数据外壳、补 Volume、检查 OHLC 与索引，再保存 broker 的 partial。每次 run() 才创建相互隔离的运行期对象。策略参数必须先是类属性，_check_params 才允许覆盖，拼写错误会尽早暴露。

Strategy.I(func, ...) 在 init() 中立即使用全量数据计算，检查结果是一至二维且最后一维与数据等长，再包装为携带名称、索引和绘图元数据的 _Indicator。主循环缩短的是可见切片，不是逐 bar 重算函数。

for i in range(1 + warmup, len(original_data)):
    data._set_length(i + 1)
    strategy.indicator = indicator[..., :i + 1]
    broker.next()
    strategy.next()
观察点：start = 1 + _indicator_warmup_nbars(strategy)。+1 保证至少可以安全访问上一条记录。

自测

类属性 n = 10 时，run(n=20) 会覆盖声明参数；run(nn=20) 因未知参数失败。哪些动作每次 run 一次，哪些每 bar 一次？

装配、参数校验、init()、指标全量计算和统计各一次；窗口推进、指标切片、broker 撮合、strategy 决策按 bar 重复。

阶段2 · 时间推进与未来数据隔离

底层数组完整，不等于策略能看到未来

读完要能区分 _set_length、_update 与 warmup，并指出可绕过的未来数据泄漏。

三种职责

_Data._set_length

只改变逻辑长度并清空列切片缓存。下次访问 data.Close 才按需返回当前前缀，不复制整张 DataFrame。

_Data._update

init() 后重建完整列数组，以接纳策略对 data.df 的列修改；它不是时钟推进。

_Indicator 切片

主循环把完整指标替换为 [..., :i + 1]，保持数据窗口和策略指标同长。

warmup

对非 scatter 指标取前置 NaN 长度的最大值，决定第一个策略调用位置。

安全路径与绕过路径

正常通过 self.data、_Array.s/.df 与 self.I() 属性访问时，只能看到当前窗口。但若在 init() 保存当时的全长 self.data.Close 引用，并在 next() 直接读取其尾部，就绕过了切片协议。这是框架约定，不是进程级沙箱。

精确时间边界：Buy & Hold 从 Close[warmup] 而主循环从 1+warmup 开始。若 warmup=9，基准从 Close[9] 起算；主循环从 i=10 开始，第一次 Strategy.next() 看见 11 条记录，因此基准起点比首次策略调用早一根 bar。

_Data.__get_array() 缓存当前长度的切片视图。若推进长度后不清缓存，策略会继续拿到旧前缀；若直接缓存全长，又会破坏可见性。

尚未标记

阶段3 · 订单、交易与持仓状态模型

Order 是意图，Trade 是 lot，Position 是视图

读完要能解释关闭 API、反向下单、hedging 与 exclusive_orders 的差别。

Order

等待撮合的指令。size 正负代表方向；0 到 1 之间的绝对值表示可用流动性比例，绝对值至少 1 时是整数单位。

Trade

独立成交 lot，保存 entry/exit price、bar、size、tag、commission 及可选 SL/TP 关联。

Position

动态汇总 broker 上的活动 trades，不独立保存仓位。多个 trade 仍只有一个净视图。

SL / TP

都是带 parent trade 的反向 contingent Order：SL 是 stop-market，TP 是 limit。

生命周期

  1. Pending Order
  2. Active Trade
  3. 部分 reduce 或 close order
  4. Closed Trade

Trade.close(portion) 创建带 parent_trade 的反向市价单并插入队首；Position.close 对每个活动 trade 分别做同样动作。普通 sell() 是独立订单：hedging=False 时可 FIFO 抵消 long，hedging=True 时多空可并存。

exclusive_orders=True 会取消非 contingent 挂单，并给现有 trades 插入 close orders，再排入新订单。父 trade 全平时，SL/TP 同步取消形成 OCO；部分平仓会复制一份 closed trade 记录，并缩小剩余 trade 与附属单 size。

sell 没有指定 parent trade，其结果受 hedging、FIFO 抵消、单位换算和可用 margin 影响。若要关闭指定 lot，应使用 Trade.close()；关闭净视图用 Position.close()。

阶段4 · 精读撮合引擎

_process_orders 把一根 OHLC 变成确定性模型结果

读完要能推导 market、limit、stop-market、stop-limit，并标记同 bar 先后未知的边界。

  1. 取当前 Open / High / Low
  2. 检查 stop
  3. 检查 limit 或 market
  4. 处理 parent / FIFO
  5. 检查 margin
  6. open / reduce / close
订单触发基础成交价
默认市价单下一次 broker tick通常当前 Open
trade_on_close=True 的非 contingent 纯市价单仍在下一次 broker tick前一 bar Close，时间索引也记前一 bar
long stop-marketHigh ≥ stopmax(Open, stop)，跳空可劣化
long limitLow ≤ limitmin(Open, limit)，开盘可改善
contingent SL / TPSL 用 stop，TP 用 limit关闭价不再做独立 entry spread 调整

账户与成本

  • spread 进入方向调整后的 stand-alone 入场价：long × (1 + spread)，short × (1 - spread)。
  • commission 是现金流，开仓和平仓各收一次；可以是比例、固定加比例或函数。
  • margin 转成 leverage = 1 / margin，约束可新增名义价值，不是简单扣掉一笔保证金现金。
  • equity = cash + 活动 trades 的未实现 PnL；margin available = max(0, equity - margin used)。
同 bar 歧义：OHLC 只能证明价格范围，不能证明先到 SL 还是 TP。market entry 新附属单会在同 bar 重处理，SL 因队列在前形成保守结果；stop/limit 父订单的不可判定附属单可被警告并延后。没有订单簿、容量、冲击、排队和逐笔路径。

stop 命中，基础成交价是 max(108, 105) = 108，体现向上跳空后以更差价格开多。之后再计算方向 spread 与 commission。

阶段5 · 结果、统计与绘图

交易层与权益层使用不同、但彼此一致的事实源

读完要能从 _Stats 反向追溯任一主要指标的数据来源。

  1. broker._equity
  2. broker.closed_trades
  3. compute_stats
  4. _Stats
  5. _plotting.plot

broker 每次 next() 后写当期 equity。循环结束，前缀与空位被补齐，compute_stats 同时收到完整 equity、closed trades、原 OHLC 和策略。

必须分清：交易层指标只用 closed trades，而权益类指标用完整 equity。具体地，_trades、胜率、SQN、平均交易等只使用 broker.closed_trades；Equity Final、Return、回撤等读取完整 equity，其中仍含活动 trade 的未实现盈亏。finalize_trades 决定末尾活动 trade 是否被强制关闭并进入交易表与交易层指标。

compute_stats 从交易构造 Size、Entry/ExitBar、价格、PnL、Commission、ReturnPct、时间、Duration、Tag 与指标值；从 equity 构造收益、累计峰值、DrawdownPct 和回撤持续时间。SQN 是交易 PnL 的函数，因此不直接依赖 OHLC；年化、Sharpe、Sortino、Calmar 则还依赖时间索引、交易日假设和重采样。

_Stats 只是改变显示行为的 pd.Series 子类，并包含 _strategy、_equity_curve、_trades 三个内部结果。绘图读取并可重采样副本，不写回 broker。

Buy & Hold 边界：收益从 Close[warmup] 起算；主循环从 1 + warmup 开始。这不是数据第一行，也不是首次 Strategy.next() 所在 bar。

Max. Drawdown → dd.max() → 1 - equity / cumulative max(equity) → broker._equity → cash + unrealized PnL。

阶段6 · 参数优化与并行化

SharedMemoryManager 复制数据，Pool 分发参数 batch

读完要能画出主进程、共享内存、worker 和最佳参数复跑的边界。

搜索方式

  • grid：参数笛卡尔积先过 constraint；max_tries=None 穷举，比例或正整数转换为抽样概率，所以随机网格次数是近似值。
  • SAMBO：模型驱动搜索，以负的 maximize(stats) 为最小化目标并缓存结果；默认最多 200 次，不走 grid 的共享内存 worker 路径。
  1. 组合 + constraint
  2. 生成参数 batch
  3. df2shm 复制数据
  4. Pool.imap
  5. worker 精简 stats
  6. 主进程最佳参数复跑
精确复制语义：共享内存按参数 batch 复制。grid 优化中，每生成一个参数 batch，就调用一次 smm.df2shm(self._data)，把 index 和各列复制到该 batch 的一组新共享内存段。共享内存避免的是 batch 内每个参数组合分别重复序列化 DataFrame，并非整个 optimize 只复制一次。

主进程只向 worker 发送共享内存 name/shape/dtype 元数据和参数 batch。Backtest._mp_task 重建只读 DataFrame，逐组 run()；有交易时只返回不以下划线开头的公开统计，无交易返回 None，最后只 close worker 句柄。创建者在退出时 close + unlink。

backtesting.Pool 是可覆盖入口；spawn 默认路径会警告并退化为线程池。MultiBacktest 在多份数据上独立运行同一策略，每份都有独立现金、broker 与统计；它不是共享资金、多资产 margin 或组合风控引擎。

过滤 _strategy、_equity_curve 和 _trades 可显著减小跨进程传输；选出最佳组合后，主进程再完整 run 一次取得最终对象。

交易生命周期推演

10 单位 long：从订单到净 PnL 79.45

参数：cash=10,000；size=10；spread=1%；commission=0.5%；margin=1；trade_on_close=False；SL=95；TP=110。

1 · 排队2 · 开仓3 · 持有4 · TP 平仓

步骤 1：bar 1 决策，只排队

OHLC=100/102/99/100。Strategy.next() 调用 buy(size=10, sl=95, tp=110)。新 Order 入队，当前策略调用不成交。

步骤 2：bar 2 先 broker 撮合

OHLC=100/108/99/106。基础 Open=100；long spread 后 EntryPrice=101。开仓 commission=10 × 101 × 0.5% = 5.05，cash=9,994.95；创建 Trade 与两张 size=-10 的 SL/TP 单。

步骤 3：同 bar 检查与持有

Low=99、High=108，SL 95 和 TP 110 均未触发。Close=106 时未实现 PnL=10 × (106 - 101) = 50，equity=10,044.95。

步骤 4：bar 3 命中 TP

OHLC=109/112/104/111。TP limit=110 命中，以 110 平仓；退出 commission=10 × 110 × 0.5% = 5.50 并取消 SL。毛 PnL=90，总 commission=10.55，净 PnL=79.45，最终 cash/equity=10,079.45。

步骤 1 / 4：排队
SL 分支：若下一 bar Open=94、High=97、Low=92，关闭 long 的 short stop-market 使用 min(Open, stop)=94，成交价比触发价 95 更差；统计仍保留原 SL=95。触发价与成交价不可混为一谈。

系统边界 · 从研究回测到实盘

保留清晰边界，不复制研究模型的全部假设

Backtesting.py 适合紧凑的单标的逐 bar 研究内核，但不是完整实盘系统。MultiBacktest 也只是独立数据集比较包装。

数据

现有：输入 OHLC(V)。新增：采集、版本化、复权、时区/日历、质量检测。

撮合与执行

现有：bar OHLC 确定性触发。新增：订单簿、逐笔、容量、排队、部分成交、冲击、券商网关与对账。

状态与组合

现有：单次运行内存状态。新增：持久化、恢复、事件日志、共享现金、多资产净额、组合 margin 与风险预算。

风控与运维

新增：事前/事后风控、敞口限制、熔断、限频、监控、告警、权限、密钥管理与审计。

研究治理

新增：训练/验证/样本外、走步分析、多重检验控制和实验追踪。

分层原则

Strategy 表意图；Clock 控因果；Broker 明确成交假设；Portfolio/Accounting 拥有真相；Reporting 只消费事件。

订单簿需求不能只给 _process_orders 加一个参数；共享资金也不能把 MultiBacktest 改几行就得到。二者都会改变事件、状态所有权与时间粒度。
